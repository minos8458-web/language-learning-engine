## 1. 목적

이 문서는 다음 두 가지만을 정의한다.

- 엔진이 "학습자가 이 문법을 습득했다"고 판단하기 위해 **어떤 상태·신호를 추적해야 하는가** (WHAT)
- 복습·생성 엔진이 **반드시 지켜야 할 원칙**은 무엇인가 (WHAT)

이 문서는 알고리즘(간격 반복 공식, 정확한 임계값 등)을 정의하지 않는다. 그것은 HOW 단계이며, 이 문서가 정의한 제약을 만족하는 한도 내에서 이후 엔진 설계 문서가 결정한다.

## 2. 문법 습득 상태 모델 (Grammar Acquisition State Model)

문법 노드의 습득 상태는 아래 6단계로 정의한다. 이 상태는 **문법 노드 자체의 속성이 아니라, "사용자 × 문법 노드" 조합에 귀속되는 진행 데이터**임을 명확히 한다. 이는 4장 제약 요약에서 Schema에 대한 필수 요구사항으로 다시 명시한다.

| 단계 | 정의 | 진입 조건 | 엔진에서 허용되는 것 |
|---|---|---|---|
| **Not Introduced** | 사용자에게 노출된 적 없음 | 초기 상태 | 존재만 함. 노출·생성 어디에도 사용 불가 |
| **Introduced** | 명시적 학습을 통해 소개됨 | 명시적 학습 이벤트 발생 (6장 참조) | 노출 가능. 아직 인출 연습 없음 |
| **Studying** | 인출 연습이 시작됨, 정확도 불안정 | Introduced 이후 최초 Active Recall 시도 | Active Recall 연습 대상 |
| **Practicing** | 반복 연습 중, 정확도 상승 추세 | 일정 수준의 누적 정답률 도달 | Interleaving 대상, 간격 반복 스케줄 진입 |
| **Mastered** | 인출이 정확하고 여러 세션에 걸쳐 안정적 | 5장의 Accuracy/Confidence 기준 충족 | AI 생성 재료로 사용 가능 여부는 **7장에서 결정 필요** |
| **Automatic** | 정확 + 신속 + 무의식적 사용 | Mastered + Response Time 기준 충족 (5장) | 자유로운 조합 생성 재료, 복습 빈도 최소화 |

**상태 전이는 단방향이 아니다.** 복습 실패, 장기 미사용 등으로 상태는 하락할 수 있다 (예: Automatic → Practicing). 정확한 하락 조건은 HOW 단계(엔진 설계)에서 정의하되, "상태는 퇴행 가능해야 한다"는 것은 이 문서가 강제하는 제약이다.

## 3. Universal Grammar Concepts (범언어 문법 개념)

### 3.1 왜 필요한가

베트남어의 "đã"와 영어의 "-ed"는 표면 형태는 다르지만 같은 기능(과거 시제)을 수행한다. 언어별 문법을 서로 완전히 독립된 노드로만 다루면, Language Pack을 확장할 때마다 Grammar Graph를 처음부터 다시 설계해야 한다.

이를 막기 위해 LLE는 **언어에 종속되지 않는 상위 개념 레이어**를 먼저 정의한다.

### 3.2 구조

```
Universal Concept (언어 무관, 엔진 코어에 1회 정의)
        │
        │  "표현한다 (expressed_by)"
        ▼
언어별 Grammar Node (Language Pack에 정의)
```

- **Universal Concept**: 시제, 상(相), 양태, 조건, 부정, 비교, 수량 등 기능적 범주. 예: `PAST`, `PROGRESSIVE`, `ABILITY`, `CONDITIONAL`, `NEGATION`, `COMPARISON`, `QUANTITY`.
- **언어별 Grammar Node**: 특정 언어가 해당 개념을 표현하는 구체적 문법 형태. 예: 베트남어 `đã` → `PAST`, 베트남어 `đang` → `PROGRESSIVE`.
- 하나의 Grammar Node가 여러 Concept에 연결될 수도 있다 (예: 완료상은 `PAST` + `PERFECT` 조합으로 표현되는 언어가 있음).

### 3.3 이것이 가능하게 하는 것

- **Language Pack 확장의 구조적 재사용**: 새 언어를 추가할 때 Universal Concept 레이어는 그대로 두고, "이 언어는 이 개념을 어떻게 표현하는가"만 매핑하면 된다.
- **개념 기반 커리큘럼 설계**: "이 언어는 조건법을 접속법 형태로 표현한다"처럼, 언어 간 구조적 차이를 설계 단계에서 명시적으로 다룰 수 있다.
- **의미 있는 Interleaving**: 표면 형태가 아니라 기능(개념) 단위로 교차 연습을 구성할 수 있다.

## 4. Review Principles (복습 원칙)

복습 엔진은 아래 원칙을 예외 없이 만족해야 한다. 정확한 계산식은 HOW 단계에서 정의한다.

1. **간격 확장 원칙** — 상태가 Studying → Automatic 방향으로 진행할수록 복습 간격은 확장되어야 한다.
2. **실패 시 축소·연쇄 원칙** — 인출 실패는 해당 노드의 간격을 축소시킬 뿐 아니라, Grammar Graph 상에서 **선행(prerequisite) 관계로 연결된 노드**의 복습도 함께 트리거할 수 있어야 한다. (→ Graph에 prerequisite 엣지 타입 필요, 4장 요약 참조)
3. **교차 강제 원칙 (Interleaving Mandate)** — 하나의 복습 세션은 동일 노드를 반복 제시하지 않는다. 여러 노드·여러 Universal Concept를 섞어 구성해야 한다.
4. **능동 인출 전용 원칙** — 복습은 항상 사용자가 먼저 인출을 시도한 뒤에만 정답/피드백을 제공한다. 정답을 먼저 보여주는 방식은 복습으로 인정하지 않는다.
5. **표층 변주 원칙** — 동일 문법을 반복 복습할 때 어휘·문맥은 매번 달라져야 한다. 동일 문장을 그대로 재사용하는 것은 금지한다 (패턴 암기 방지, PROJECT_VISION 1.1(4) 대응).
6. **전이 검증 원칙** — 일정 주기로, 학습 당시 접하지 않은 새로운 문맥에서 해당 문법을 사용하는 문제를 포함해야 한다. 이는 암기와 습득을 구분하는 핵심 장치다.

## 5. Mastery 판정 모델

### 5.1 정확도만으로는 부족한 이유

정답률만으로 Mastered를 판정하면, 짧은 시간에 반복 시도해서 우연히 맞춘 경우와 실제로 자동화된 경우를 구분할 수 없다. LLE는 세 가지 신호를 함께 본다.

| 신호 | 의미 | 역할 |
|---|---|---|
| **Accuracy** | 인출 시도 대비 정답 비율 | 기본 정확성 지표 |
| **Confidence** | 사용자가 답을 낼 때의 확신 정도 | 우연한 정답과 진짜 습득을 구분 |
| **Response Time** | 인출에 걸린 시간 (문항 난이도로 정규화) | 자동화(무의식적 사용) 여부 판단 — Automatic 상태 진입의 핵심 근거 |

### 5.2 결합 방식 (개념적 정의)

- **Mastered 진입**: 여러 번의 간격이 벌어진 복습에 걸쳐 Accuracy와 Confidence가 모두 높게, 그리고 안정적으로 유지되어야 한다. 단발성 고득점은 인정하지 않는다.
- **Automatic 진입**: Mastered 조건 + Response Time이 해당 문법 난이도의 기준선 이하로 떨어져야 한다.
- 정확한 임계값, 가중치, 정규화 방식은 HOW 단계(엔진 알고리즘 설계)에서 정의한다. 이 문서는 "세 신호가 모두 필요하다"는 것과 "정확도 단독으로 상태를 결정하지 않는다"는 것만을 제약으로 강제한다.

### 5.3 Confidence 측정 방식 (결정 완료)

Confidence는 **행동 추론을 기본으로, 자기보고를 보정용 보조 지표로 사용하는 병행 모델**로 확정한다.

- **행동 추론 (기본, 매 시도마다 자동 계산)**: 응답 시간, 수정 횟수, 힌트 사용 여부, 직전 연속 정답 수 등 행동 데이터를 기반으로 한다.
- **자기보고 (보조, 주요 상태 전이 시점에만 수집)**: Studying 종료, Practicing 종료 등 주요 학습 단계 전환 시점에 사용자의 자기평가를 받아, 행동 추론값과 비교·보정한다.
- **채택 이유**: 자기보고만 쓰면 사용자의 주관적 과신·과소평가에 취약하고, 행동 추론만 쓰면 실제 심리적 확신 상태를 놓칠 수 있다. 상시(행동) + 체크포인트(자기보고)로 병행하면 객관성과 주관적 신뢰도를 함께 반영하면서도, 매 문항마다 자기보고를 요구해 학습 흐름을 끊는 문제를 피할 수 있다.

## 6. 명시적 학습 게이팅 원칙 (Explicit Study Gating Principle)

- **Not Introduced → Introduced 전이는 오직 "명시적 학습 이벤트"로만 발생한다.** 생성 연습 문장 속에 우연히 포함된 문법에 노출되는 것(부수적 노출, incidental exposure)은 이 전이를 유발하지 않는다.
- **명시적 학습 이벤트**란 사용자가 해당 문법 노드에 대한 설명/레슨을 능동적으로 완료한 행위를 말한다. (정확한 UX 형태는 HOW/UI 단계에서 결정)
- 이 원칙의 존재 이유: PROJECT_VISION의 comprehensible input 제약을 지키기 위해서다. AI가 "언젠가 자연스럽게 익히겠지"라며 미도입 문법을 생성물에 흘려 넣는 것을 구조적으로 차단한다.

## 7. AI 생성 제약 (AI Generation Constraints)

- AI가 새로운 문제·문장·회화를 생성할 때 사용할 수 있는 문법 노드의 최소 자격 상태는 **Practicing 이상**으로 확정한다. (Practicing, Mastered, Automatic 노드만 생성 재료로 사용 가능. Not Introduced / Introduced / Studying 노드는 생성에 사용 금지.)
- **조합 규칙 (확정)**: 생성물이 둘 이상의 Grammar Node를 조합하는 경우, **조합에 포함된 노드 전원이 개별적으로 Practicing 이상**이어야 한다. 하나라도 Studying 이하인 노드가 섞여 있으면 그 조합 전체를 생성에 사용할 수 없다. "평균적으로 기준을 넘으면 된다"거나 "일부만 충족하면 된다"는 식의 부분 허용은 인정하지 않는다.
- **채택 이유**: Mastered 이상으로 제한하면 comprehensible input 원칙은 가장 안전하게 지켜지지만, 초반 학습자는 조합 가능한 재료가 거의 없어 Generation Practice가 실행될 기회가 늦게 열린다. Practicing 이상을 기준으로 하면 아직 완전히 안정화되지 않았지만 인출 가능한 문법까지 포함되어, 더 이른 시점부터 생성 연습이 가능해지고 문법 자동화 촉진이라는 프로젝트 핵심 목적에 부합한다. 조합 규칙을 "전원 충족"으로 엄격히 정한 것은, 학습자가 아직 인출하지 못하는 문법이 결과물에 섞여 들어가 comprehensible input을 깨는 상황을 원천 차단하기 위함이다.
- **후속 설계 참고 (HOW 단계, 지금 결정하지 않음)**: 모든 노드가 Practicing 이상이라는 조건을 만족해도, Practicing 노드끼리만 여러 개 조합되면 문장 전체의 안정성이 낮을 수 있다. 조합 내 Practicing 노드 비율을 제한하는 규칙이 필요한지는 Grammar Graph/생성 엔진 설계 문서에서 다룬다.

## 8. Schema/Graph 설계 제약 요약

이 문서로부터 GRAMMAR_SCHEMA.md와 Grammar Graph 설계가 반드시 만족해야 할 제약을 정리한다.

**확정된 제약 (MUST)**

- [C1] 습득 상태(2장)는 **문법 노드의 정적 정의가 아니라 사용자별 진행 데이터**로 저장되어야 한다. Schema는 "정적 문법 노드"와 "사용자 진행 레코드"를 별도 엔터티로 분리해야 한다.
- [C2] 습득 상태는 6단계이며 퇴행(하락)이 가능해야 한다.
- [C3] Universal Concept는 언어 독립적인 별도 최상위 엔터티로 존재해야 하며, 언어별 Grammar Node는 반드시 1개 이상의 Universal Concept를 참조해야 한다 (`concept_ids` 필수 필드).
- [C4] Grammar Graph는 최소 두 종류의 엣지를 지원해야 한다: 일반 연결 관계, 그리고 **prerequisite(선행) 관계** — 복습 실패 연쇄(4장 원칙 2)에 사용됨.
- [C5] 사용자 진행 레코드는 Accuracy, Confidence, Response Time을 노드별로 각각 저장해야 한다 (단일 mastery boolean 금지).
- [C6] 진행 레코드는 "명시적 학습 이벤트 발생 여부"를 노출 횟수와 별도로 구분해서 저장해야 한다 (6장).
- [C7] 복습 대상 선정 로직은 항상 다중 노드를 교차 샘플링할 수 있는 구조여야 한다 (단일 노드 반복 제시를 원천적으로 배제하는 구조).
- [C8] 사용자 진행 레코드는 시도(attempt)마다 행동 기반 Confidence 계산에 필요한 원시 신호(응답 시간, 수정 횟수, 힌트 사용 여부, 직전 연속 정답 수)를 저장해야 한다.
- [C9] 사용자 진행 레코드는 주요 상태 전이 시점에 수집되는 자기보고 Confidence 값과, 이를 행동 추론값과 비교한 보정(calibration) 결과를 별도로 저장해야 한다.
- [C10] 생성 엔진의 노드 필터링 로직은 상태가 **Practicing 이상**인 노드만 생성 후보로 선택할 수 있어야 한다 (Not Introduced / Introduced / Studying 노드는 항상 제외). 둘 이상의 노드를 조합할 때는 **조합에 포함된 노드 전원이 개별적으로 Practicing 이상**이어야 하며, 일부 노드만 기준을 충족하는 부분 허용은 금지한다.
