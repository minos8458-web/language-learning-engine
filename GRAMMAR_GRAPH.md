## 2. Graph 구조

- **Grammar Graph = { Grammar Node } + { Grammar Relation }.** 이는 별도로 저장되는 자료구조가 아니라, GRAMMAR_SCHEMA.md에 정의된 Relation 레코드 위에서 조회 시점에 구성되는 **뷰(view)**다.
- **상태가 입혀진 그래프(State-Colored Graph)**: 동일한 추상 그래프에, 요청 시점의 사용자 Progress 데이터를 오버레이하여 "이 사용자에게는 이 노드가 Practicing" 같은 사용자별 그래프를 구성한다. 이 오버레이는 **조회 시점에 계산되는 것이며, 사용자별 그래프 사본을 저장하지 않는다** (저장 시 사용자 수 × 그래프 크기로 폭증하는 것을 방지).
- **서브그래프**: 언어별 서브그래프(`language` 일치), Concept별 서브그래프(`concept_ids` 공유) 등은 모두 조회 조건으로 구성되는 뷰이며 별도 저장 대상이 아니다.

## 3. Graph 탐색 알고리즘

- **선행 탐색(Prerequisite Search)**: 노드 A로부터 시작해 A가 의존하는 선행 노드들을 찾는다(`from_node_id = A`인 PREREQUISITE 관계를 따라 `to_node_id`를 수집, 필요 시 전이적으로 확장). **Review Cascade Engine(5장)의 핵심 연산이다.**
- **후행 탐색(Dependent Search)**: 노드 A로부터 시작해 A를 선행 조건으로 요구하는 노드들을 찾는다(`to_node_id = A`인 관계를 따라 `from_node_id`를 수집). 커리큘럼 설계 시 "이 노드를 아직 배우지 않으면 어떤 노드들이 막히는가"를 파악하는 데 쓰인다.
- **순환 방지 검증**: GRAMMAR_SCHEMA §6에 따라 `PREREQUISITE`는 항상 `UNIDIRECTIONAL`이어야 한다. 이 문서는 한 걸음 더 나아가, **PREREQUISITE 관계로만 이루어진 부분 그래프가 순환 없는 DAG여야 한다**는 것을 Language Pack 배포 전 검증 절차로 강제한다(런타임이 아니라 배포 전 정적 검증).
- **Concept-Node 정합성 검증**: GRAMMAR_SCHEMA §6의 정합성 제약("Concept 레벨 선행관계는 Node 레벨에도 반영되어야 한다")을 실제로 점검하는 절차. 역시 Language Pack 배포 전 검증 단계에서 수행한다.
- **가중 탐색**: 여러 경로·후보가 있을 때 `weight`가 높은 Relation을 우선 채택한다(Review Cascade 우선순위, Interleaving 샘플링에서 활용).

## 4. Learning Flow Engine (핵심 장)

### 4.1 전체 흐름 — 한 문법 노드의 일생

```
NOT_INTRODUCED
   │ [명시적 학습 이벤트]
   ▼
INTRODUCED
   │ [최초 능동 인출 시도]
   ▼
STUDYING
   │ [반복 연습, 정답률 상승]
   ▼
PRACTICING
   │ [Mastery 판정 통과: Accuracy+Confidence 안정]
   ▼
MASTERED
   │ [Response Time 기준 통과]
   ▼
AUTOMATIC
```

각 화살표는 Progress Engine(8장)이 기록하는 이벤트로만 트리거된다. 어느 단계에서든 인출 실패는 아래 4.2의 **실수 처리 루프**로 진입한다.

### 4.2 실수 처리 루프 (Mistake Handling Loop)

단순 오답 처리로 끝나지 않는다. 아래 6단계를 고정한다.

1. **오답 발생**: 사용자의 인출 시도가 기대 정답과 불일치. AttemptRecord에 `is_correct=false`로 기록(GRAMMAR_SCHEMA §5).
2. **원인 분류**: 이 오답의 원인을 분류한다. *(분류 세분화 수준은 4.2.1에서 결정)*
3. **관련 Grammar Node 확인**: 시도된 문항이 대상으로 한 노드(들)를 특정하고, 2번의 분류 결과가 다른 노드를 가리키는지(예: 표면적으로는 어순 오류지만 실제로는 다른 문법 노드의 미숙 때문) 확인한다.
4. **선행 Prerequisite 탐색**: 3번에서 확인된 노드(들)의 `PREREQUISITE`를 선행 탐색하여(3장 알고리즘), 오류의 근본 원인이 더 앞선 선행 노드의 미습득에 있는지 확인한다.
5. **Review Cascade 생성**: 2~4단계에서 식별된 노드 집합을 Review Cascade Engine(5장)에 전달해 복습 대상 목록을 산출한다.
6. **재시도 또는 보류**: 원인이 단순 실수(다른 노드와 무관)로 판단되면 즉시 재시도를 제공한다. 근본 원인이 선행 노드 미습득으로 판단되면, 현재 노드 재시도는 보류하고 선행 노드 복습을 먼저 배치한다.

#### 4.2.1 원인 분류 체계 (확정)

**최상위 분류를 `SELF`(자체 미숙) / `TRANSFER`(다른 노드로 인한 전이 오류) 이분법으로 확정한다.** MVP에서는 이 두 축만 사용해 엔진을 단순하게 유지한다.

**확장 가능한 구조**: 이 분류는 최종 분류가 아니라 최상위(top-level) 분류로 설계한다. 향후 `SELF` 하위에 `KNOWLEDGE`/`RECALL` 등, `TRANSFER` 하위에 `COMBINATION`/`CONTEXT` 등 세부 유형이 필요해지면, 이미 준비된 `error_subcategory` 필드(GRAMMAR_SCHEMA §5 v1.3에 추가)에 값을 채우기 시작하면 된다 — **필드 구조 자체를 나중에 바꿀 필요가 없다.**

| | 안 A(이분법) | 안 B(다범주) | 안 C(분류 생략) |
|---|---|---|---|
| 채택 여부 | **채택 — MVP 최상위 분류** | 미채택(향후 하위 분류로 흡수 가능) | 미채택 |

**채택 이유**: 파일럿 단계의 핵심 목표는 "이 오류가 선행 노드 문제인지 아닌지"를 가려내는 것이며, 이것이 Cascade의 존재 이유(4.2의 4~6단계)를 그대로 충족한다. 다범주 분류에 필요한 정교한 자동 진단 로직은 지금 근거 데이터가 없으므로, 상위 구조는 지금 정하되 세부 값은 나중에 채우는 방식으로 설계했다.

**사용 규칙**: `error_category = TRANSFER`인 시도만 4단계(선행 Prerequisite 탐색)로 진행한다. `error_category = SELF`인 시도는 그 탐색을 생략하고 곧바로 현재 노드의 반복 연습으로 처리한다(6단계로 직행).

### 4.3 상태 퇴행

반복적인 실수 처리 루프 진입(예: 짧은 기간 내 동일 노드에서 다수 실패)은 상태 퇴행을 트리거할 수 있다(LEARNING_THEORY §2). 정확한 퇴행 조건(몇 회 실패 시 몇 단계 하락)은 HOW 단계에서 정의한다.

### 4.4 다른 엔진과의 접점

| 단계 | 접점 |
|---|---|
| 명시적 학습 이벤트 | Content 엔터티(GRAMMAR_SCHEMA §3)에서 `EXPLANATION` 조회 |
| Practicing 진입 이후 | AI Generation Engine(6장)이 생성 연습 기회 제공 |
| 모든 상태 전이 | 반드시 Progress Engine(8장)을 통해서만 기록 |

## 5. Review Cascade Engine

### 5.1 트리거 조건

- 실수 처리 루프 5단계에서 호출
- 간격 반복 스케줄에 따른 정기 복습 시점 도달(LEARNING_THEORY §4 원칙 1)

### 5.2 Cascade 대상 선정 범위 (확정)

**최대 2-hop 제한으로 확정한다.** 실패 노드의 `PREREQUISITE`를 선행 탐색할 때, 직접 선행(1-hop)보다는 넓게 보되 전이적 폐쇄(전체)처럼 과도하게 확장하지 않아 복습 폭주를 막는다.

| | 안 A(1-hop) | 안 B(전체 폐쇄) | 안 C(2-hop 제한) |
|---|---|---|---|
| 채택 여부 | 미채택 — 근본 원인이 2단계 이상 위일 때 놓침 | 미채택 — 그래프가 클수록 계산 비용 증가, 관련성 낮은 노드까지 끌려올 위험 | **채택** |

**구조적 설계 방침**: 탐색 깊이는 하드코딩된 상수가 아니라 `max_cascade_depth`(기본값 2)라는 **조정 가능한 파라미터**로 설계한다. 이는 MVP 이후 데이터가 쌓이면 사용자 숙련도나 오류 유형(4.2.1의 `error_category`/`error_subcategory`)에 따라 깊이를 동적으로 조절하는 구조로 발전할 수 있는 여지를 남겨둔다. 다만 그 동적 조절 로직 자체는 지금 정의하지 않고 HOW 단계(엔진 구현)로 넘긴다 — 이 문서가 확정하는 것은 "깊이는 파라미터화되어야 하며 기본값은 2"라는 것뿐이다.

### 5.3 산출물

- 우선순위가 매겨진 복습 대상 노드 목록(`weight` 활용)
- 각 대상에 "왜 복습 대상인지"(원인 노드와의 관계) 근거를 함께 기록 — 디버깅·향후 설명 가능성 확보

### 5.4 LEARNING_THEORY §4 원칙 대응표

| 원칙 | Cascade Engine 구현 |
|---|---|
| 간격 확장 | 상태가 높을수록 다음 복습까지 간격 확장(함수는 HOW 단계) |
| 실패 시 축소·연쇄 | 5.2의 선행 탐색 로직 |
| 교차 강제 | Cascade 산출물은 항상 다중 노드 목록이며 Interleaving Engine(7장)에 그대로 전달 |
| 능동 인출 전용 | Cascade 문항은 항상 정답 비노출 상태로 생성 |
| 표층 변주 | 재복습 시 이전과 다른 `surface_form`/예문을 쓰도록 AI Generation Engine에 요청 |
| 전이 검증 | 주기적으로 이전에 없던 문맥의 문항을 섞음 |

## 6. AI Generation Engine

### 6.1 필터 파이프라인 (순서 고정, 절대 변경 불가)

1. **[최우선] 학습자 수준 초과 금지 필터**: 후보를 `state IN (PRACTICING, MASTERED, AUTOMATIC)`으로 제한(GRAMMAR_SCHEMA §7). 조합 시 **전원**이 이 기준을 충족해야 한다. **문장이 아무리 자연스럽거나 흥미로워도, 이 필터를 통과하지 못하면 생성 후보에서 완전히 제외한다.** 자연스러움·흥미도는 이 필터를 통과한 후보 안에서만 비교 대상이 된다.
2. **Concept 목표 필터** (선택): `target_concept_id` 지정 시, 해당 Concept을 구현하는 노드를 최소 1개 포함하는 조합만 남김(GRAMMAR_SCHEMA §8).
3. **Interleaving 협조 필터**: 최근 세션에서 과반복된 조합을 후순위로 내림.
4. **표층 변주 필터**: 동일 노드의 이전 `surface_form`/예문 재사용 회피.
5. **품질 스코어링**: 1~4를 통과한 후보 중에서만 자연스러움·다양성을 비교한다.

**순서가 고정되는 이유**: 1번이 나중에 오면 "자연스러운 문장을 먼저 고르고 나서 검열"하는 흐름이 되어, comprehensible input 원칙이 사후 필터링에 의존하게 된다. 1번을 최우선으로 고정해야 애초에 안전하지 않은 후보가 파이프라인에 진입조차 하지 않는다.

### 6.2 생성 난이도 자동 조절 체계 (확정)

AI Generation Engine의 목표는 "생성 가능/불가능"의 이분법이 아니다. **학습 흐름을 끊지 않는 것**이 목표이므로, 후보가 부족한 상황은 "실패"가 아니라 **난이도를 단계적으로 낮추는 자동 조절 과정**으로 정의한다.

**4단계 자동 조절 사다리**

1. **조합 문제 생성** (기본 경로): 6.1 필터 파이프라인을 통과한 후보들로 둘 이상의 노드를 조합해 생성한다.
2. **단일 Grammar Node 문제 생성**: 1번이 불가능하면(조합 가능한 후보 2개 미만), Practicing 이상 노드가 하나라도 있을 때 그 노드 하나만으로 최소 단위 문제를 생성한다. 난이도는 내려가도 **안전 기준(Practicing 이상)은 내려가지 않는다.**
3. **사전 제작 콘텐츠 사용**: 2번도 불가능하면(Practicing 이상 노드가 0개, 완전 초심자 구간), Content 엔터티(GRAMMAR_SCHEMA §3)의 `EXAMPLE` 중 현재 학습 중이거나 방금 `Introduced`된 노드에 연결된 것을 대신 제공한다. 이는 AI 생성물이 아니라 사전 검수된 콘텐츠이므로 6.1 필터 적용 대상이 아니다.
4. **사용자 안내**: 1~3 모두 불가능한 경우(해당 노드에 사전 제작 콘텐츠조차 없는 경우)에만 상황을 안내한다. 이는 정상 경로가 아니라 **콘텐츠 공백 신호**이며 Language Pack 제작 프로세스에 피드백되어야 한다.

**세 대안의 통합**: 애초 비교했던 세 안은 서로 배타적 선택지가 아니라, 이 사다리 안에서 각각 제자리를 찾는다 — 단일 노드 완화(2단계), 사전 제작 콘텐츠(3단계), 사용자 안내(4단계, 최후 수단).

**설계 원칙**: 각 단계는 상위 단계가 실패했을 때만 순차적으로 시도하며 임의로 건너뛰지 않는다. 사용자 입장에서는 조합 문제든 단일 노드 문제든 사전 제작 예문이든 모두 **자신의 현재 수준에 맞게 자동 조절된 정상적인 학습 콘텐츠**로 인식되어야 하며, 어느 단계도 "오류 화면"으로 노출되지 않는다.

### 6.3 ALTERNATIVE 관계의 활용

하나의 조합이 필터를 통과했을 때, 동일 기능을 수행하는 `ALTERNATIVE` 관계의 다른 노드로 치환한 변형도 후보에 포함할 수 있다 — 표층 변주 원칙을 지원한다.

---

## 7. Interleaving Engine

- **입력**: Review Cascade 산출물(5장) 또는 신규 연습 세트
- **규칙**:
  - 동일 노드 연속 배치 금지
  - `CONTRAST` 관계로 연결된 노드 쌍은 의도적으로 같은 세션에 포함(대조 학습 유도)
  - 동일 Concept Category 노드가 세션 내 과반을 넘지 않도록 분산
- **산출물**: 순서가 부여된 문항 시퀀스

---

## 8. Progress Engine

- 모든 상태 전이, Confidence 갱신(행동 추론 + 자기보고 보정), Coverage/Depth 계산(CONCEPT_SCHEMA §10)은 **이 엔진을 통해서만** 이루어진다.
- **단일 쓰기 경로 원칙**: 이 엔진만이 GRAMMAR_SCHEMA §5의 필드에 쓰기(write)할 수 있다. 다른 어떤 엔진(Learning Flow, Review Cascade, AI Generation, Interleaving)도 Progress 데이터를 직접 기록하지 않는다 — 반드시 Progress Engine에 위임한다. 이는 상태 불일치(두 엔진이 같은 값을 다르게 기록하는 사고)를 원천 차단하기 위한 아키텍처 제약이다.
- **제공 기능(조회 인터페이스)**: 상태 조회, AttemptRecord 기록, Confidence 보정 계산, Concept Coverage/Depth 계산, 다음 복습 시점 계산.

---

## 9. Graph Optimization

- **원칙**: 최적화는 계산 결과(WHAT)를 절대 바꾸지 않고 속도(HOW)만 개선한다. 예컨대 Coverage/Depth 캐싱은 허용되지만, 캐싱으로 인해 실시간 계산과 다른 값이 나온다면 그 전략은 기각한다.
- 언어팩이 늘어날 때 인덱스는 언어별로 분리되어, 한 언어의 그래프 크기 증가가 다른 언어의 조회 성능에 영향을 주지 않아야 한다(확장성 우선순위, PROJECT_VISION §2.2).
- 정확한 캐싱·인덱싱 기법은 HOW 단계(엔진 구현 문서)에서 정의한다.

---

## 10. 금지 사항

- Grammar Graph를 Relation 레코드와 별도로 중복 저장하는 것 (2장 위반)
- 사용자별 그래프 사본을 영구 저장하는 것 (2장의 "오버레이는 조회 시점 계산" 원칙 위반)
- 실수 처리 루프를 생략하고 단순 정답/오답만 기록하는 것 (4.2 위반)
- AI Generation Engine에서 필터 순서를 바꿔 품질 스코어링을 최우선 필터보다 먼저 적용하는 것 (6.1 위반)
- Progress Engine을 거치지 않고 다른 엔진이 Progress 데이터를 직접 기록하는 것 (8장 위반)
- `PREREQUISITE` 관계에 순환이 존재하는 상태로 Language Pack을 배포하는 것 (3장 위반)
- 최적화를 이유로 Coverage/Depth 등 계산 결과값을 근사치로 영구 대체하는 것 (9장 위반)
