# PROJECT_VISION.md
## Language Learning Engine (LLE) — 프로젝트 헌법

> 이 문서는 LLE의 모든 설계·구현 결정의 최상위 판단 기준이다.
> 새로운 기능, 새로운 언어 팩, 새로운 UI가 이 문서와 충돌한다면, 바뀌어야 하는 것은 이 문서가 아니라 그 기능이다.

---

## 0. 문서의 지위

이 문서는 스펙(spec)이 아니라 헌법(constitution)이다.
스펙은 "무엇을 만들 것인가"를 정의하지만, 헌법은 "무엇을 만들지 말아야 하는가"와 "왜 만드는가"를 정의한다.

LLE의 모든 하위 문서(아키텍처 문서, 데이터 스키마, Grammar Graph 명세, Language Pack 가이드 등)는 이 문서에서 파생되며, 이 문서와 모순될 수 없다.

이 문서를 수정하려면 6장의 의사결정 원칙을 따라야 한다.

---

## 1. WHY — 우리는 왜 이것을 만드는가

### 1.1 기존 언어 학습 앱의 구조적 한계

시중의 언어 학습 앱(Duolingo류를 포함한 대다수)은 공통적으로 다음과 같은 구조적 한계를 가진다.

**(1) 게임화가 목적이 되어버린 설계**
스트릭, XP, 리더보드, 레벨업 같은 외적 동기 장치가 학습 설계의 중심을 차지한다. 사용자는 "오늘도 스트릭을 유지했다"는 성취감을 얻지만, 그것이 실제 회화 능력과 비례한다는 보장이 없다. 게임화는 리텐션(retention) 지표는 개선하지만, 언어 습득(acquisition) 지표와는 별개로 움직인다.

**(2) 문법의 고립화**
대부분의 앱은 문법을 "레슨 1, 레슨 2, 레슨 3"처럼 순차적이고 고립된 단위로 가르친다. 레슨 5에서 배운 문법과 레슨 12에서 배운 문법을 조합해서 새로운 문장을 만들어야 하는 상황이 설계에 포함되지 않는다. 그 결과, 학습자는 개별 문법 규칙은 "알지만" 그것을 실제 발화에서 "조합해서 쓰는" 능력, 즉 전이(transfer)에 실패한다.

**(3) 단어 암기 중심의 착시**
어휘 카드, 매칭 게임, 빈칸 채우기 등 대부분의 반복 학습 콘텐츠가 어휘량 확장에 최적화되어 있다. 그러나 회화 능력의 병목은 대개 어휘가 아니라 문법 자동화(문법을 생각하지 않고 즉각적으로 정확하게 사용하는 능력) 부족이다.

**(4) 정적 콘텐츠와 생성 연습의 부재**
문제와 문장이 사전에 제작되어 고정되어 있다. 같은 문제를 반복하면 학습자는 "문제를 기억"하게 되며, 이는 언어 능력이 아니라 패턴 암기다. 학습자가 배운 문법 재료로 스스로 새로운 문장을 생성해보는 연습(generation practice)이 구조적으로 부족하다.

**(5) 개인화의 부재**
모든 학습자가 동일한 커리큘럼과 동일한 순서를 따른다. 학습자마다 다른 문법 습득 상태, 다른 취약점을 반영한 맞춤형 복습·생성이 이루어지지 않는다.

**(6) 회화 능력과 앱 성과 지표의 괴리**
앱 내부의 성공 지표(DAU, 스트릭 유지율, 레슨 완료율)가 실제 언어 능력 향상과 직접적으로 연결되어 있다는 근거가 약하다. 앱은 "학습하는 느낌"을 최적화하고, 실제 "학습 효과"는 부차적이다.

### 1.2 LLE가 해결하려는 핵심 문제

LLE는 위 한계의 근본 원인을 다음과 같이 정의한다.

> **문법이 서로 연결되지 않은 채로 가르쳐지기 때문에, 학습자는 문법을 "아는" 상태에서 "자동화된" 상태로 넘어가지 못한다.**

이를 해결하기 위해 LLE는 세 가지 문제에 집중한다.

1. **문법 자동화(Grammar Automation)의 부재**
   — 개별 문법을 반복 노출시키는 것이 아니라, 이미 학습한 문법들을 조합하여 사용하는 훈련이 필요하다.

2. **암기와 체화 사이의 간극**
   — "안다"와 "쓸 수 있다" 사이의 간극을 좁히는 것은 노출 빈도가 아니라 인출(retrieval)과 생성(generation)의 반복이다.

3. **AI 생성 콘텐츠의 무통제 확장 문제**
   — AI가 학습자 수준을 넘어서는 문법/어휘를 사용해 문장을 생성하면, comprehensible input 원칙이 깨지고 학습자는 좌절하거나 암기로 회피한다. AI의 생성 범위는 반드시 "학습자가 이미 습득한 문법 노드"로 제한되어야 한다.

## 2. WHAT — LLE의 핵심 철학

### 2.1 우리는 게임이 아니라 엔진을 만든다

LLE의 산출물은 "베트남어를 배우는 재미있는 앱"이 아니다. 산출물은 **어떤 언어든 투입하면 문법 자동화 학습 경험을 생성할 수 있는 엔진**이다. 베트남어는 이 엔진의 첫 번째 검증 사례(pilot)일 뿐, 최종 목적지가 아니다.

이 구분은 사소하지 않다. "앱을 만든다"는 사고는 UI와 콘텐츠 제작에 자원을 집중시키지만, "엔진을 만든다"는 사고는 구조와 데이터 모델에 자원을 집중시킨다. LLE는 후자를 택한다.

### 2.2 절대 우선순위

모든 설계 논의에서 충돌이 발생하면 아래 순서가 우선한다. 상위 항목이 하위 항목을 위해 희생되는 일은 없다.

| 순위 | 항목 | 비고 |
|---|---|---|
| 1 | 학습 효과 | 모든 것의 존재 이유 |
| 2 | 교육 설계 | 학습 효과를 구현하는 방법론 |
| 3 | 시스템 아키텍처 | 교육 설계를 지탱하는 구조 |
| 4 | 데이터 구조 | 아키텍처를 표현하는 형식 |
| 5 | 확장성 | 다른 언어로 확장 가능한가 |
| 6 | 유지보수성 | 장기적으로 관리 가능한가 |
| 7 | UI | 마지막 단계, 학습 설계를 침해할 수 없음 |

### 2.3 기능 필터: "이것이 회화 능력을 향상시키는가?"

새로운 기능을 제안하기 전에 반드시 이 질문을 통과해야 한다.

> **"이 기능이 실제 회화 능력 향상에 도움이 되는가?"**

대답이 "아니오" 또는 "불확실"이면 그 기능은 제안되지 않는다. "사용자가 좋아할 것 같다", "다른 앱에도 있다", "재미있을 것 같다"는 이 질문에 대한 답이 아니다.

### 2.4 다섯 가지 학습 원리

LLE의 모든 학습 메커니즘은 아래 다섯 원리 중 최소 하나 이상을 명시적으로 구현해야 한다. 어느 원리에도 해당하지 않는 기능은 존재 근거가 없다.

- **Active Recall (능동적 인출)** — 정답을 보여주는 것이 아니라 학습자가 스스로 인출하도록 요구한다.
- **Spaced Repetition (간격 반복)** — 문법 노드별 습득 상태에 따라 복습 시점이 개인화된다.
- **Interleaving (교차 학습)** — 유사한 문법을 몰아서 연습하지 않고, 이미 배운 여러 문법을 섞어서 연습한다.
- **Elaboration (정교화)** — 단순 정답 여부가 아니라 "왜 이 문법이 여기 쓰이는가"를 설명하거나 연결짓게 한다.
- **Generation Practice (생성 연습)** — 학습자가 선택지를 고르는 것이 아니라 스스로 문장/발화를 만들어내도록 요구한다.

### 2.5 Grammar Graph — 프로젝트의 핵심 구조

Grammar Graph는 LLE의 심장이다. 문법은 독립된 목록이 아니라 **노드(문법 단위)와 엣지(선행 관계·조합 가능성)로 이루어진 그래프**로 표현된다.

이 구조가 가능하게 하는 것:

- AI는 학습자가 **이미 습득한 노드 집합**만을 사용하여 새로운 문제, 새로운 문장, 새로운 회화를 생성한다. 이는 comprehensible input 원칙을 구조적으로 강제한다.
- 문법 간 **조합 가능성**이 그래프 상에 명시되므로, "따로는 알지만 같이 쓰지 못하는" 문제를 학습 설계 단계에서 발견할 수 있다.
- 새로운 언어(Language Pack)를 추가할 때, 콘텐츠가 아니라 **그래프 자체**를 이식하는 방식으로 확장한다.

### 2.6 데이터와 로직의 분리, JSON 기반 설계

모든 학습 콘텐츠, 문법 노드, 사용자 진행 상태는 JSON으로 표현되며, 이를 처리하는 로직(엔진)과 철저히 분리된다. 이는 다음을 보장하기 위함이다.

- 베트남어 데이터를 제거하고 다른 언어 데이터를 넣어도 엔진 코드는 변하지 않아야 한다.
- 데이터 오류와 로직 오류를 구분해서 진단할 수 있어야 한다.
- Language Pack 제작자가 코드를 몰라도 데이터만으로 언어를 추가할 수 있어야 한다(장기 목표).

## 3. 성공 기준 (Success Criteria)

LLE는 "출시했다"가 아니라 "학습 효과가 검증되었다"를 성공으로 정의한다. 베트남어 파일럿은 아래 기준을 통과해야 다음 언어로 확장할 자격을 얻는다.

1. **문법 노드 습득 지속률** — 특정 문법 노드를 "습득 완료"로 표시한 후, 일정 기간(예: 30일) 뒤에도 해당 문법을 정확히 인출·생성할 수 있는 비율.
2. **조합 생성 능력** — 두 개 이상의 학습한 문법 노드를 조합한 새로운 문장을 학습자가 스스로 생성할 수 있는가 (단순 인식이 아닌 생성 기준).
3. **전이 성공률** — 연습에서 다룬 적 없는 새로운 문맥에서도 동일 문법을 올바르게 적용하는 비율.
4. **반응 자동화 지표** — 문법 사용까지 걸리는 반응 시간이 학습 진행에 따라 유의미하게 감소하는가 (숙고형 사용 → 자동화된 사용으로의 전환 신호).
5. **구조적 확장 가능성** — 베트남어에서 검증된 Grammar Graph 구조와 엔진 로직이 코드 변경 없이 데이터 교체만으로 다른 언어에 적용 가능한가.

이 중 1번 항목은 게임화 지표(스트릭, DAU 등)로 대체될 수 없다. 게임화 지표는 참고 자료일 뿐 성공 판정 기준이 아니다.

## 4. 하지 않을 것 (Anti-Goals)

LLE는 명시적으로 아래를 목표로 삼지 않는다. 이는 향후 "이런 기능도 있으면 좋지 않을까"라는 제안이 들어왔을 때 판단 근거가 된다.

- **게이미피케이션을 핵심 동력으로 삼지 않는다.** 스트릭, 리더보드, 배지는 부가 기능일 수 있으나, 학습 설계의 중심이 되어서는 안 된다.
- **단어장 앱이 되지 않는다.** 어휘 확장은 문법 자동화를 지원하는 보조 수단이지, 프로젝트의 목적이 아니다.
- **정적 콘텐츠 소비 앱이 되지 않는다.** 학습자가 만들어진 콘텐츠를 소비만 하는 구조는 지양하며, 항상 생성·인출을 요구한다.
- **특정 언어에 종속된 하드코딩을 만들지 않는다.** 베트남어의 특수성(성조, 분류사 등)을 처리하기 위해 엔진 코어에 베트남어 전용 로직을 심지 않는다. 언어 특수성은 반드시 Language Pack 데이터/규칙 레이어에서 처리한다.
- **UI 트렌드를 쫓아 학습 설계를 희생하지 않는다.** 우선순위 표(2.2)에서 UI는 항상 마지막이다.
- **검증되지 않은 상태로 조기에 다국어 확장을 하지 않는다.** 베트남어에서 성공 기준(3장)을 통과하기 전에는 두 번째 언어를 시작하지 않는다.
- **AI가 학습자 수준을 초과하는 문법/어휘로 콘텐츠를 생성하도록 허용하지 않는다.** Grammar Graph의 습득 노드 제약을 우회하는 생성은 금지한다.

## 5. 장기 비전 (Long-Term Vision)

1. **1단계 — 베트남어 파일럿**: Grammar Graph, 다섯 학습 원리, JSON 데이터 구조를 베트남어로 구현하고, 3장의 성공 기준으로 검증한다.
2. **2단계 — 엔진과 데이터의 완전한 분리 검증**: 베트남어 데이터를 제거했을 때 엔진 코드가 무의미해지지 않고, 다른 언어 데이터를 넣었을 때 코드 수정 없이 동작하는지 확인한다.
3. **3단계 — Language Pack 표준화**: 두 번째 언어를 데이터 팩 형태로 추가하며, 이 과정에서 드러나는 엔진의 숨은 가정(베트남어에 암묵적으로 의존하던 부분)을 제거한다.
4. **4단계 — 범용 문법 자동화 엔진**: 최소한의 언어학적 데이터 입력(문법 노드, 예문, 관계)만으로 새로운 언어의 학습 경험을 생성할 수 있는 수준에 도달한다.
5. **궁극적 지향점**: LLE는 특정 언어의 학습 앱이 아니라, "인간이 문법을 자동화하는 과정" 자체를 모델링하고 지원하는 범용 엔진으로 남는다.

## 6. 의사결정 원칙 (Decision-Making Principles)

이 문서 및 LLE의 모든 하위 설계는 아래 원칙을 따라 결정된다.

1. **사고 순서는 항상 WHY → WHAT → HOW다.** 구현(HOW)부터 논의하지 않는다.
2. **불확실한 것은 추측으로 결정하지 않는다.** 근거가 불충분하면 결정을 보류하고 확인을 요청한다.
3. **복수의 설계안이 가능하면 장단점을 비교 제시하고, 승인 후 진행한다.** 임의로 하나를 선택하지 않는다.
4. **구조가 우선이고 구현은 나중이다.** 문제가 발견되면 즉시 코드로 해결하지 않고, 더 나은 구조를 먼저 제안한다.
5. **단순성 > 확장성 > 유지보수성 순으로 구조적 트레이드오프를 판단한다.** 단, 이는 2.2의 절대 우선순위(학습 효과 등)보다 하위에 있다.
6. **중요한 결정에는 반드시 이유를 문서화한다.** "왜 이렇게 했는가"가 기록되지 않은 결정은 잠정적인 것으로 간주하고 재검토 대상이 된다.
7. **이 문서와 충돌하는 제안은 기각한다.** 이 문서를 개정하려면 별도의 명시적 승인 절차를 거쳐야 한다.
